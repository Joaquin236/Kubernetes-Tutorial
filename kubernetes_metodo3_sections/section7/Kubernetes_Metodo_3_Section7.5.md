## 65.1º Las autorizaciones gestionan los que pueden entrar el cluster y lo que tienen permitido. El administrador puede realizar todas las operaciones de comandos: agregar, consultar, modificar y borrar. el desarrollador puede consultar pero no puede borrar un componente del servicio. Los bots no pueden realizar acciones sin autorización, se necesita el modo de requisito mínimo de autoriaciones.

## 65.2º El mecanismo del Node lo compone el administrador, el kube-api y el kubelet con fines de gestión. Las opciones de consultar y escribir las autoriza el node-authorizer.

## 65.3º El mecanismo del ABAC lo compone el usuario de desarrollo, este usuario puede leer, crear y borrar los pods del cluster. Se puede crear un fichero con formato json de las políticas de autorización. [{"kind":"Policy","spec":{"user":"dev-user","namespace":"*","resource":"pods","apiGroup":"*"}}]. Este fichero lo recibe el servidor de api. A medida que se añaden más usuarios, se necesita adaptar el fichero json y crear nuevas entradas con el nombre correspondiente. Los controles de acceso basados en funciones facilitan la administración.

## 65.4º El grupo de desarrollo tiene 3 usuarios con la autorización de administrar los pods, mientras el grupo de seguirad tiene 1 usuario y solo puede crear y consultar el sistema de los certificados CSR. 

## 65.5º El mecanismo Webhook lo compone un usuario, el kube-api y un agente de políticas, si el agente de políticas es uno externo, se puede configurar para delegar trabajos sin necesidad de hacerlos localmente en la infraestructura directa. El agente verifica el estado de la autorización y realiza la acción necesaria.

## 65.6º También tenemos los mecanimos ["AllwayAllow","AllwaysDeny"] que establecen peticiones siempre permitidas y/o siempre denegadas sin realizar la verificación de autenticación. El fichero /usr/local/bin/kube-apiserver tiene configurado las opciones para realizar operaciones siempre permitidas o denegadas. El parámetro --authorization-mode=["Parameter"]. Puede tener cualquiera de las opciones disponibles, para hacerlo más restrictivo le cambiamos por Node,RBAC,Webhook. El sistema solo autorizará las peticiones de los servicios mencionados en la lista de parámetros, denegando lo demás. Cuando se deniega una solicitud se envía al otro nodo y hasta llegar al nodo final. Cuando sea aprobado la solicitud no se realizan más verificaciones y el permiso es concedido.
